GENERAL INFORMATION:

 

What is NESmaker?

NES maker is a software for PC that allows users to create brand new, hardware playable, cartridge based games for the Nintendo Entertainment System in a modern development environment.  These games will play on any hardware based system that can play NES games, and on any NES emulator that supports mapper 30.


Why did you decide to make NESmaker?

That question may be best answered by checking out The New 8-bit Heroes on Amazon...it's kind of a long, interesting story, and we hope you enjoy it!  If you're here, it stands to reason that you would appreciate the film.

 

 

How does NESmaker work?

Games for the Nintendo Entertainment System were originally written in a programming language called 6502 Assembly.  When run through a program known as an assembler, the low-level program language is converted into hexedecimal instructions and values that are accessed directly by the NES to perform various functions.

 

Games made with NESmaker are made exactly the same way, but without the user ever having to write any code.  Under the hood are tens of thousands of lines of pre-written, interchangeable, modifiable assembly code.  When the user makes changes in the NESmakerâ€™s intuitive graphic user interface, the underlying assembly code is updated based on those changes.  And what's great is that as you become more comfortable with NESmaker, you have full control over the code underneath and can begin writing your own to extend the tool's capabilities.

 

Choosing to export the game runs the code through the ASM6 assembler and automatically produces a NES ROM file, which opens in an emulator of your choosing (must be compatible with mapper 30).  Choosing the make-cart option exports the game and then flashes it to a mapper 30 cartridge via a Kazzo cartridge flashing device, at which time it is playable on any hardware based, NES compatible system.  To find out more about flashing hardware or flashable cartridges, please visit our website.

 

 

Is NESmaker easy to use?

NESmaker is comparable in function to other modern game development environments such as GameMaker, RPG Maker, Unity, Construct, and other similar programs.  It is designed so that beginners with zero experience can easily learn the basic workflow.  And while the quality of game you can make is still largely dependent on your creativity and understanding of good game design, NESmaker will allow you to skip much of the challenging barriers of technical understanding that have prevented many developers from creating experiences for this particilar console.

 

 

What kind of games can I make with NESmaker?

NESmaker is capable of making games of all types.  Within the NES constraints, any type of game is theoretically possible.  Keep in mind, NESmaker functions in conjunction with the underlying assembly code, much of which is interchangeable and modifiable.  What we provide are component pieces.  We give you the basic building blocks, and you determine how to put them together.  Also, as the community for NESmaker grows, so will the library of available scripts and tutorials.

 

 

INTERFACE OVERVIEW:

 

 

Menu Options

 

The FIle Menu

 

NEW

When you select NEW you are creating a brand new project.  There, you will have the opportunity to utilize or create a graphics assets folder.  You have three options.

  1.  USE EXISTING TILESETS: Choosing this option will set the project's tileset to the default tiles in the Graphic Assets folder.
  2. CREATE BLANK TILES: This will create a new folder with a library of blank tiles for a fresh start.
  3. CREATE DEFAULT TILES: This will create a new folder, but will copy the default project's tiles into it instead of creating blank tiles.


In most cases, users are likely to create blank tilesets for new projects.

 

LOAD

This will load a previously saved file.  The extension for NESmaker files is MST (Mystic Searches Tool, which was the impetus for this project).

 

SAVE

This saves the current project.  If it has not been saved yet, use Save As.

 

SAVE AS

This allows you to name and determine where to save your MST project.

 

EXPORT
This allows you to copy the GameEngineData folder, which hold all of the game's scripts, to another location.  We originally used this when versioning, which is now somewhat redundant, but we left it in in case anyone had a use for this.

 

 

The Project Menu

 

INFO

 

Initial Settings Tab

This window allows you to set some initialization data.  Much of what is found in this window can be set in different parts of the tool, however the Game Timer Speed and the ticker that determines whether or not you begin above ground can be set exclusively here.  The tickers for bosses defeated, relics, songs, and items are holdovers from Mystic Searches and are currently unused, however their bit data still exports so it could be used for various things.

 

Triggered Screen Types Tab

The engine allows for 256 different screen types.  Here, you can set these screen types to be triggered as part of initialization, which is great for testing how a screen functions normal versus triggered.

 

SCREEN GROUPS

This is for use with the map.  You can use these labels and color coding to help you organize your game world.  This is not data that gets output for anything, it's for tool organization only.

 

PROJECT SETTINGS

The Project Settings window really allows you to customize the tool for your specific game.  There are a lot of ways in which it does this.

 

Labels

You can change the labels that you see for assets here in the labels tab.

 

Tilesets

You can change the path to included tilesets and change their display name.

 

Emulator

Here, you can set the emulator that you want NESmaker to launch exported games in for testing (must be able to run Mapper30).

 

Script Settings

Here, you can define the scripts do use for certain functions.  For instance, you can determine what 16 monster actions your game uses.  These script definitions are the main part of module creation.

 

 

IMPORT PROJECT MODULE

This imports .mod files, which are a collection script definitions and select variables.

 

EXPORT PROJECT MODULE

This exports the current collection of script definitions and select variables to a .mod file.

 

 

The Test Menu

 

EXPORT

This exports the game data.

 

EXPORT AND TEST

This exports the game data and opens it in the emulator that is defined in Project Settings.

 

RETEST LAST EXPORT

This is used to A/B a new export versus the previous export.

 

MAKE CART

This is used to flash the game directly to a mapper30 cartridge using a Kazzo flasher.  This requires the latest flasher and drivers from Infinite Nes Lives to function.  See our site for details.

 

 

Maps

In NESmaker, you have two different maps, each with up to 256 screens.  In the asset tree, they are listed as overworld and underworld, but this is just a result of how they were used in Mystic Searches.  In reality, the screens on these maps can be used in many different ways.  They do not have to represent overworld and underworld at all.

 

A new menu option appears if you are in map view.  In the MAP menu, there are four items.

 

 

The Map Menu

 

EXPORT BOTH MAPS

This exports the data contained in both maps into a .nesmap file.

 

IMPORT BOTH MAPS

This imports .nesmap files and uses it to fill both maps.  Keep in mind, if your assets have changed, maps may also be different.

 

EXPORT MAP IMAGE

This exports an image file of the current map.

 

ERASE MAP

This erases the current map and sets all map values to zero.

 

 

Other Map Functionality

 

COPY AND PASTE SCREENS

You can copy and paste a screen by clicking on it, pressing control-C on your keyboard, moving to another empty screen, and clicking control-V on your keyboard.

 

RIGHT CLICK

You can right click on a screen to access a few more options from a floating menu.  You can delete the screen, you can set a comment, a screen group, set a screen as the starting screen (which you can also do in several places in the tool), and you can view latest export to see the collision table data in code.

 

SHOW GROUPS / SHOW TOOLTIPS

Screen Groups don't translate to any output data for the game (screen types are handled in the Project Settings), but it can be helpful for organization.  If you have selected a group for a screen and put a comment, clicking Show Tooltips will reveal that comment on mouse-over.  Show Types will put a different color around screens based on their type.  These are great organizational tools you can use while building screens.

 

 

Screen Editor

When creating a new screen, you set the bank to the one which contains the graphics that you want to use.  Double clicking on a screen will open that screen for editing.

 

TILESETS AND PALETTES

On the left of the Screen Editor, you can see the loaded tilesets and palettes that are loaded for the current screen.  You can show or hide an overlay of the HUD.  You can switch back and forth to see the day palette versus the night palette.  And you can edit the palettes directly by right clicking on the colors.

 

ASSETS

Here, you see the available assets for this screen.  The available assets are automatically generated, determined by which tilesets you have selected.  You can create groups within the assets, which help keep them organized.  To create an asset group, click Add Group and give it a name.  To add an asset to a group, right click on the asset and assign it to a group.  To see only particular groups, go to the dropdown at the top of the asset column and select that group.  To get back to seeing all assets available, choose All assets from the dropdown.

 

THE SCREEN PAINTER

In the screen painter, your mouse position will show a ghosted image of the selected asset.  Clicking the mouse will place the graphics, collision data, and attribute data to that position on the screen.

 

KEYBOARD SHORTCUTS

  • ESCAPE: deselects current asset.
  • Left, Right, Up, Down: move to adjacent screen.
  • TAB: moves to the same screen on opposite map.
  • + and - : zoom in and out of the screen and allow you to see more or less of ghosted image of adjacent screens.  This is very helpful in lining up collisions with adjacent screens.
  • Q,W,E,R: These keys repaint the tile under the mouse with respective sub palettes.  Q paints with subpal0, W paints with subpal1, E paints with subpal2, and R paints with subpal3.
  • 1,2,3,4: These keys paint autopaths.  See section about paths to understand how paths work.
  • SHIFT: Shifts assets larger than 16x16 so that the handle of the asset is on the right instead of left.  This is great for placing objects on the left of the screen.
  • CRTL: Shifts assets larger than 16x16 so that the handle of the asset is on the bottom instead of the top.  This is great for placing objects on the top of the screen.
  • 5: The 5 key draws tile00, attribute00, and collision type 0, essentially acting like an eraser.
  • 6: The 6 key draws "blank tiles" so you do not need to waste a tileset space on an all blank 16x16 tile.

 

TILE DATA READOUT

At the top of the screen, you can see information about the tile that the mouse is hovering over.  It shows you tile number, subpalette ID, and collision information.

 

SCREEN INFO

Here you can set details about the screen.

  • Screen Flags:  These are changeable in name, and you can use them in your scripts to change the behavior of the screen.  For instance, you could make a bit flag to make a room a boss room, and then update your code to read that bit to do boss room things.
  • Path Groups: You can set the path group that the screen uses.
  • Warp In Screen Location: This determines where on the screen an object will appear when it enters with a warp-type transition.
  • Warp Out Screen: This defines the map screen to warp to for warp tiles.
  • Warp Out To Underworld: Ticking this will cause the object to warp to the second 'underworld' map.
  • Screen Speed: currently unused.
  • Screen Type: Used with triggers.  Trigger code allows you to flip the trigger bit for a screen type.  This is where you set that screen type.
  • Songs: Select songs for each state a screen can be in.
  • NPC Indexes: The NPC groups that NPCs on this screen use.
  • Monsters tabs: Here you can set the monster bank, tileset, and group this screen will use.  Also, you can determine what monster palette will be used for this screen.  You can set all four screen states independently.

 

HUD and Boxes

NESmaker currently makes use of four boxes.  By default, each serves its own purpose.

  • HUD AREA - designates the area of the HUD.
  • Playable Area - sets the bounds for a screen.
  • Box 1, Text Box Area - this is the box that is used for the default textbox.  Currently, text box areas must line up in a particular way on the grid for them to restore correctly when hidden.  If you notice that a when hiding a text box, it leaves the wrong color information on its right or bottom edge, either shrink or grow the box by one tile to fix the problem.  This is a bug we are aware of.  Also, while you can overlap boxes, it is inadvisable at this time to place the textbox to overlap the HUD, as it will restore the underlying graphic values and not the hud values when the box gets hidden.
  • Box 2, Monster spawn area - This box allows you to set an area where randomly placed monsters spawn.  It is useful in making sure monsters don't spawn too close to the edge, colliding with an entering player upon screen load.
  • Box 3, User - you can use these box defines for any project specific needs you might have.

 

By setting the demo screen, you are able to view the box placements as compared to actual game screens.  To set a box, simply click and drag your mouse, and once you have the desired area selected, hit one of the *set* buttons, and that area will then be defined for that type of box.

WRITING CUSTOM CODE:

 

ASM Basics

 

Built In Functions

The ASM6 code assembler allows for functions called macros.  A macro is a single instruction that can be placed in code that can substitute for a longer set of instructions, making code easier to read and write.  Essentially, we have created some baseline macros that work with the existing NESmaker scripts that will all you to do common behaviors with more streamlined code.

 

The challenge with this is that as we refine the modules, and while users begin to create their own, some functions may change or no longer be relevant.  We will try to keep this list as organized as possible so that it is easy to find updates and changes, but a good understanding of the base code that the macro represents will be very helpful.

 

To use a macro, you simply write the macro code and include the arguments that it needs.  Arguments are specifications that the code needs to run.  For instance, if you had a macro called LoadScreen, the routine may need to know what number screen you are trying to load.  This might be held in an argument.  LoadScreen #$07 (macro argument) might load screen number 7 while LoadScreen temp would load the screen that corresponds with the value being held in the temp variable.

 

Some macros have many arguments needed to define them, some don't need any at all.

 

 

 

CreateObject arg0, arg1, arg2, arg3

 

     arg0 = x value

     arg1 = y value

     arg2 = object type

     arg3 = action step

 

DeactivateCurrentObject

 

ChangeObjectState arg0, arg1

      arg0 = what state to change the object to

      arg1 = initial animation timer

 

 

StartMoving arg0, arg1

      arg0 = object to move

      arg1 = direction

 

This macro uses the built in physics found in default physics scripts. It acceleration and deceleration as long as bounds and tile collision. It is not a direct movement, but rather turns on a check for whether or not the object should move.

 

Keep in mind, this does not alter the facing-direction of the object, just the actual motion.

 

Some constant definitions for direction are:

     MOVE_RIGHT = #%11000000

     MOVE_LEFT  = #%10000000

     MOVE_DOWN = #%00110000

     MOVE_UP = #%00100000

 

     MOVE_RIGHT_DOWN = #%11110000

     MOVE_LEFT_DOWN  = #%10110000

     MOVE_RIGHT_UP  = #%11100000

     MOVE_LEFT_UP  = #%10100000

 

 

StopMoving arg0, arg1

     arg0 = object to stop moving

     arg1 = stop moving which direction?
